# 아껴맵 프로젝트 리서치

## 문서 목적

이 문서는 아껴맵 프로젝트의 목적, 도메인 방향, MVP 범위, 구현 제약사항을 정리하는 리서치 문서입니다.

상세 구현 순서, 생성할 파일 목록, 패키지 구조, 테스트 계획은 별도 문서인 `docs/plans/PLACE_DOMAIN_PLAN.md`에서 다룹니다. 이 문서에서는 개발자가 방향을 잃지 않도록 핵심 원칙과 우선순위만 정리합니다.

## 현재 저장소 상태

현재 저장소는 기획과 문서 중심의 초기 단계입니다.

```text
akkyomap/
├─ AGENTS.md
├─ README.md
└─ docs/
   ├─ 01_requirements.md
   ├─ 02_erd.md
   ├─ 03_api_spec.md
   ├─ plans/
   └─ research/
      └─ PROJECT_RESEARCH.md
```

아직 `backend/`와 `frontend/`는 생성되어 있지 않습니다. 현재 단계에서는 먼저 도메인 방향과 개발 범위를 명확히 한 뒤, 이후 계획 문서에서 백엔드 구현 순서를 구체화하는 것이 적절합니다.

## 프로젝트 목적

아껴맵은 대학생과 취업준비생이 학교 주변의 생활비 절약 장소를 지도 기반으로 찾고 공유할 수 있게 하는 서비스입니다.

대상 장소는 다음과 같습니다.

- 저렴한 식당
- 카페
- 무료 또는 저렴한 학습 공간
- 프린트/복사 가능 장소
- 생활 편의시설

이 프로젝트의 핵심은 단순 지도 CRUD가 아닙니다. 사용자가 제보한 장소 정보를 바로 노출하지 않고, 관리자 승인과 신고 처리를 거쳐 신뢰도 있는 장소 데이터로 운영하는 것입니다.

포트폴리오 관점에서 강조해야 할 방향은 다음입니다.

- 지도 기반 장소 조회: 현재 지도 범위 안의 승인된 장소만 조회하여 불필요한 응답 데이터를 줄입니다.
- 사용자 제보 신뢰도 관리: 장소 상태를 `PENDING`, `APPROVED`, `REJECTED`, `HIDDEN`으로 나누어 운영합니다.
- 운영 가능한 관리자 기능: 승인 대기 장소, 신고 장소, 숨김/복구 처리를 분리해 실제 서비스 운영 흐름을 보여줍니다.

## 핵심 사용자

### 일반 사용자

일반 사용자는 학교 주변의 저렴한 장소를 찾고, 새로운 장소를 제보하며, 잘못된 정보를 신고하는 사용자입니다.

주요 행동은 다음과 같습니다.

- 승인된 장소 목록 조회
- 지도에서 승인된 장소 마커 확인
- 장소 상세 정보와 가격 정보 확인
- 장소 제보
- 잘못된 정보 신고
- 리뷰 확인 및 작성

초기 구현에서는 로그인 없이 장소 등록, 조회, 신고가 가능하게 둘 수 있습니다. 다만 이후 JWT와 권한 분리가 도입될 가능성을 고려해 사용자 연관관계를 초기에 강하게 결합하지 않는 것이 좋습니다.

### 관리자

관리자는 사용자 제보 데이터를 검토하고 서비스에 노출할지 결정하는 운영자입니다.

주요 행동은 다음과 같습니다.

- 승인 대기 장소 목록 조회
- 장소 승인
- 장소 반려
- 장소 숨김 처리
- 숨김 장소 복구
- 신고 내역 확인

관리자 API는 초기 개발 단계에서는 인증 없이 동작할 수 있습니다. 하지만 추후 Spring Security와 JWT 적용 후 `ADMIN` 권한만 접근 가능하도록 변경해야 하며, 구현 시 이 점을 TODO로 남겨야 합니다.

## MVP 범위

아껴맵의 MVP는 한 번에 모든 기능을 구현하는 방식이 아니라, 장소 제보와 운영 승인 흐름을 중심으로 단계적으로 확장해야 합니다.

### 1차 MVP

1차 MVP는 Place 도메인의 핵심 승인 흐름을 완성하는 단계입니다.

범위는 다음과 같습니다.

- Place 등록
- 등록 시 `PENDING` 상태 저장
- 관리자 승인
- 관리자 반려
- `APPROVED` 장소 목록 조회
- 지도 마커 조회

핵심 흐름은 다음과 같습니다.

```text
사용자 장소 등록
-> PENDING 상태 저장
-> 관리자 승인 또는 반려
-> APPROVED 상태 장소만 일반 목록과 지도에 노출
```

### 2차 MVP

2차 MVP는 사용자 제보 데이터의 신뢰도를 관리하기 위한 신고와 숨김 처리 단계입니다.

범위는 다음과 같습니다.

- 신고 저장
- 관리자 숨김 처리
- 관리자 복구 처리

초기에는 신고 내역 저장과 관리자의 수동 숨김/복구를 우선합니다. 신고 3회 이상 자동 숨김은 이후 확장 기능으로 분리하는 것이 적절합니다.

### 3차 MVP

3차 MVP는 승인된 장소의 사용자 경험을 보강하는 리뷰 기능입니다.

범위는 다음과 같습니다.

- 리뷰 작성

리뷰는 `APPROVED` 상태의 장소에만 작성할 수 있어야 합니다. 승인되지 않은 장소에 리뷰가 작성되면 장소 신뢰도 관리 흐름이 깨지기 때문입니다.

## 주요 도메인

### Place

아껴맵의 중심 도메인입니다. 사용자가 제보한 장소이며, 승인 상태에 따라 노출 여부가 결정됩니다.

초기 필드는 다음을 기준으로 합니다.

```java
private Long id;
private String name;
private PlaceCategory category;
private String address;
private Double latitude;
private Double longitude;
private String priceInfo;
private String description;
private PlaceStatus status;
private LocalDateTime createdAt;
private LocalDateTime updatedAt;
```

초기에는 `User`와 강하게 연결하지 않습니다. 장소 등록, 조회, 승인 흐름을 먼저 동작시키는 것이 우선입니다.

### PlaceStatus

장소의 운영 상태를 나타냅니다.

```java
public enum PlaceStatus {
    PENDING,
    APPROVED,
    REJECTED,
    HIDDEN
}
```

상태 의미는 다음과 같습니다.

- `PENDING`: 사용자가 등록했지만 아직 관리자 승인을 받지 않은 상태
- `APPROVED`: 관리자가 승인하여 지도에 노출 가능한 상태
- `REJECTED`: 관리자가 반려한 상태
- `HIDDEN`: 신고 누적 또는 관리자 판단으로 숨김 처리된 상태

### PlaceCategory

장소의 종류를 나타냅니다.

```java
public enum PlaceCategory {
    RESTAURANT,
    CAFE,
    STUDY_SPACE,
    PRINT_COPY,
    CONVENIENCE
}
```

표시명은 프론트엔드에서 처리하거나 enum 내부 필드로 처리할 수 있습니다. 초기에는 도메인 복잡도를 줄이기 위해 enum 값만 사용해도 충분합니다.

### Review

승인된 장소에 대한 사용자 리뷰입니다.

주요 필드는 다음과 같습니다.

```java
private Long id;
private Long placeId;
private Integer rating;
private String content;
private Integer crowdedLevel;
private Boolean priceAccurate;
private LocalDate visitedDate;
private LocalDateTime createdAt;
```

리뷰는 3차 MVP에서 다룹니다.

### Report

잘못된 장소 정보나 부적절한 내용을 신고하는 도메인입니다.

신고 사유는 다음을 기준으로 합니다.

```java
public enum ReportReason {
    WRONG_PRICE,
    CLOSED_PLACE,
    INAPPROPRIATE_CONTENT,
    DUPLICATED_PLACE,
    ETC
}
```

신고 저장과 관리자 숨김/복구는 2차 MVP에서 다룹니다.

## Place 도메인의 역할

Place는 아껴맵의 핵심 운영 단위입니다. 장소 데이터 하나가 사용자 제보, 관리자 승인, 지도 노출, 리뷰, 신고 흐름의 중심이 됩니다.

Place가 담당해야 할 책임은 다음과 같습니다.

- 장소 기본 정보 보관
- 장소 카테고리 보관
- 지도 조회를 위한 좌표 보관
- 가격 정보와 설명 보관
- 현재 운영 상태 보관
- 상태 변경 행위 제공

상태 변경은 Service에서 값을 직접 바꾸는 방식보다, Place 엔티티에 명확한 메서드를 두는 방식이 적절합니다.

```java
place.approve();
place.reject();
place.hide();
place.restore();
```

이 방식은 도메인 행위가 코드에 명확하게 드러나고, 상태 변경 테스트를 작성하기도 쉽습니다.

## 관리자 승인/반려 흐름

관리자 승인/반려는 사용자 제보 데이터를 서비스에 노출할지 결정하는 핵심 운영 흐름입니다.

승인 흐름은 다음과 같습니다.

```text
1. 사용자가 장소를 등록한다.
2. 서버는 장소를 PENDING 상태로 저장한다.
3. 관리자가 승인 대기 목록을 조회한다.
4. 관리자가 장소를 승인한다.
5. 서버는 장소 상태를 APPROVED로 변경한다.
6. 일반 장소 목록과 지도 마커 API에 해당 장소가 포함된다.
```

반려 흐름은 다음과 같습니다.

```text
1. 관리자가 승인 대기 장소를 조회한다.
2. 정보가 부정확하거나 부적절한 장소를 반려한다.
3. 서버는 장소 상태를 REJECTED로 변경한다.
4. 반려된 장소는 일반 목록과 지도 마커 API에 포함되지 않는다.
```

주의할 점은 관리자 기능을 일반 사용자 API와 섞지 않는 것입니다. 관리자 API는 `/api/admin/...` 경로로 분리하고, 추후 `ADMIN` 권한 적용 예정임을 코드 주석 또는 TODO로 남겨야 합니다.

## 신고/숨김 처리 흐름

신고는 사용자 제보 데이터의 신뢰도를 보완하는 운영 기능입니다.

초기 신고 흐름은 다음과 같습니다.

```text
1. 사용자가 장소를 신고한다.
2. 서버는 신고 사유와 내용을 저장한다.
3. 관리자는 신고 내용을 확인한다.
4. 문제가 있는 장소는 HIDDEN 상태로 변경한다.
5. 문제가 해결되었거나 신고가 부적절하면 APPROVED 상태로 복구한다.
```

신고 사유는 다음 범위에서 시작합니다.

- `WRONG_PRICE`: 가격 정보 오류
- `CLOSED_PLACE`: 폐업 또는 운영하지 않는 장소
- `INAPPROPRIATE_CONTENT`: 부적절한 내용
- `DUPLICATED_PLACE`: 중복 장소
- `ETC`: 기타

신고가 3회 이상 누적되면 장소 상태를 `HIDDEN`으로 변경할 수 있습니다. 다만 초기 구현에서는 자동 숨김을 바로 넣지 않고, 먼저 신고 저장과 관리자 수동 숨김/복구를 구현하는 것이 적절합니다.

## 우선 구현해야 할 기능

우선 구현 방향은 Place 도메인의 신뢰도 관리 흐름을 작게 완성하는 것입니다.

가장 먼저 완성해야 하는 목표는 다음입니다.

```text
Place 등록 -> PENDING 저장 -> 관리자 승인/반려 -> APPROVED 장소 목록 조회 -> 지도 마커 조회
```

그 다음에는 신고 저장과 관리자 숨김/복구를 붙이고, 마지막으로 리뷰 작성 기능을 추가합니다.

상세 구현 순서, 생성할 파일, 패키지 구조, 테스트 계획은 `docs/plans/PLACE_DOMAIN_PLAN.md`에서 별도로 정리합니다.

## 후순위 기능

다음 기능은 중요한 확장 요소지만, 현재 단계에서는 후순위로 유지해야 합니다.

- 로그인/JWT
- `USER` / `ADMIN` 권한 분리
- QueryDSL 기반 동적 검색
- MCP 기반 AI 장소 검색 도구
- Docker
- CI/CD
- 즐겨찾기
- 내가 등록한 장소 조회
- 가격 변경 제보
- 거리순 정렬
- 추천순 정렬
- 신고 누적 자동 숨김

특히 인증과 권한은 서비스 운영에 필요하지만, 초기 Place 흐름을 복잡하게 만들 수 있습니다. 초기 관리자 API는 인증 없이 동작하게 둘 수 있고, 이후 보안 적용 예정임을 명시하는 방식이 적절합니다.

## 지금 당장 만들면 안 되는 기능

AGENTS.md 기준으로 다음 작업은 사용자가 명시적으로 요청하기 전까지 하지 않아야 합니다.

- 로그인/JWT를 처음부터 강하게 결합하는 것
- QueryDSL을 처음부터 도입하는 것
- MSA 구조로 프로젝트를 나누는 것
- Docker, CI/CD를 필수 전제로 만드는 것
- 지도 API 키를 코드에 직접 작성하는 것
- 관리자 기능을 일반 사용자 API와 섞는 것
- Entity를 API 응답으로 그대로 반환하는 것
- 기능이 동작하기 전 과도한 추상화를 먼저 만드는 것
- MCP 기반 AI 장소 검색 도구를 초기에 구현하는 것

## 예상되는 기술적 주의점

### Entity를 직접 응답하지 않기

API 응답에는 Entity를 직접 반환하지 않아야 합니다. Request DTO와 Response DTO를 분리해야 합니다.

### Controller에 비즈니스 로직을 두지 않기

Controller는 요청을 받고 Service를 호출하는 역할만 담당해야 합니다. 승인, 반려, 숨김, 복구 같은 상태 변경 로직은 Service와 Entity 메서드로 표현해야 합니다.

### 트랜잭션 경계 명확히 하기

상태 변경이나 저장이 필요한 메서드에는 `@Transactional`을 적용해야 합니다. 조회 전용 메서드에는 `@Transactional(readOnly = true)`를 적용하는 것이 좋습니다.

### 승인된 장소만 노출하기

일반 사용자용 목록 API와 지도 마커 API는 반드시 `APPROVED` 상태만 반환해야 합니다.

특히 다음 상태는 일반 목록과 지도에 노출되면 안 됩니다.

- `PENDING`
- `REJECTED`
- `HIDDEN`

### 지도 범위 조회 조건

지도 마커 API는 현재 지도 화면 범위 안에 있는 장소만 반환해야 합니다.

기본 조건은 다음과 같습니다.

```text
latitude between swLat and neLat
longitude between swLng and neLng
status = APPROVED
```

초기에는 Repository 메서드로 처리하고, 조건이 복잡해질 때 QueryDSL 도입을 고려하는 것이 적절합니다.

### 상태 전이 규칙

상태 변경을 무조건 허용할지, 특정 상태에서만 허용할지는 구현 전에 결정해야 합니다.

초기 기준 상태 전이는 다음 정도가 적절합니다.

```text
PENDING -> APPROVED
PENDING -> REJECTED
APPROVED -> HIDDEN
HIDDEN -> APPROVED
```

다만 실제 구현 시 이미 `REJECTED`인 장소를 승인할 수 있는지, `PENDING` 장소를 숨길 수 있는지 같은 정책을 계획 문서에서 명확히 해야 합니다.

### 신고 자동 숨김은 나중에 구현하기

신고가 3회 이상 누적되면 `HIDDEN` 처리할 수 있지만, 초기 구현에서는 자동 숨김을 바로 넣지 않는 것이 좋습니다. 먼저 신고 저장 기능을 만들고, 이후 신고 수 집계와 자동 숨김을 별도 기능으로 추가해야 합니다.

### 공통 예외 처리는 기능이 늘어난 뒤 도입하기

최종적으로는 공통 예외 처리와 공통 응답 형식이 필요합니다. 다만 초기에는 단순 예외로 시작할 수 있고, 장소 등록, 조회, 승인처럼 기능이 2개 이상 붙으면 공통 예외 응답을 도입하는 것이 좋습니다.

### 테스트 계획은 별도 문서에서 다루기

테스트는 중요하지만 이 문서에서는 방향만 다룹니다. 구체적인 테스트 대상, 테스트 클래스, 테스트 케이스는 `docs/plans/PLACE_DOMAIN_PLAN.md`에서 정리합니다.

### 프론트엔드와 지도 API 키

프론트엔드는 React 또는 Vue와 Vite 기반으로 구성할 수 있습니다. 지도는 Kakao Map API 또는 Naver Map API를 사용할 수 있습니다.

주의할 점은 지도 API 키를 코드에 직접 작성하지 않는 것입니다. 추후 프론트엔드 구현 시 `.env` 기반으로 관리해야 합니다.

### 로컬 데이터베이스 선택

로컬 개발은 PostgreSQL을 권장하지만, 빠른 테스트에는 H2를 사용할 수 있습니다. 초기 백엔드 골격에서는 H2로 빠르게 API 흐름을 검증하고, 이후 PostgreSQL 프로필을 추가하는 방식이 실용적입니다.

### README 업데이트 시점

README는 백엔드 실행 방법, 주요 API, 로컬 테스트 방법이 정리될 때 업데이트하는 것이 좋습니다. 현재는 문서와 설계 단계이므로 README를 크게 확장하기보다, 실제 실행 가능한 백엔드가 생긴 뒤 갱신하는 편이 자연스럽습니다.
